**Дисциплина:** Web-разработка

Цель — понять различия между `threading`, `multiprocessing` и `asyncio` в Python
и увидеть, для каких типов нагрузки (CPU-bound и I/O-bound) каждый подход
подходит лучше.

---

## Теоретическая основа (кратко)

| Подход | Что параллелит | GIL | Где выигрывает |
|---|---|---|---|
| **threading** | потоки в одном процессе | мешает: одновременно выполняется 1 поток с байт-кодом | **I/O-bound** (GIL освобождается при ожидании I/O) |
| **multiprocessing** | отдельные процессы | у каждого процесса свой GIL → настоящий параллелизм | **CPU-bound** (нагружает все ядра) |
| **asyncio** | корутины в одном потоке (кооперативно) | не мешает (но и не обходит) | **I/O-bound** с большим числом соединений, минимальные накладные расходы |

- **CPU-bound** — задача упирается в процессор (вычисления). Помогает только
  `multiprocessing`. `threading` и `asyncio` ускорения не дают из-за GIL /
  однопоточности.
- **I/O-bound** — задача упирается в ожидание (сеть, диск). Помогают все три, но
  `asyncio` и `threading` эффективнее и легче по ресурсам, чем `multiprocessing`.

## Задача 1. Сумма чисел от 1 до N

Считается сумма всех чисел от 1 до N. Формальная цель задания — `N = 10**13`,
но «в лоб» обычным циклом Python это ~10¹³ итераций (многие часы), поэтому `N`
вынесен в параметр (по умолчанию `10**8`). От величины N поведение подходов не
меняется — меняется только абсолютное время.

Диапазон разбивается на `workers` равных частей, каждая часть суммируется
отдельной подзадачей (поток / процесс / корутина). Корректность результата
проверяется формулой Гаусса `N·(N+1)/2`.

### Файлы
- `common.py` — разбиение диапазона, CPU-нагрузка (`partial_sum`), эталон;
- `threading_sum.py` — потоки;
- `multiprocessing_sum.py` — процессы (`multiprocessing.Pool`);
- `async_sum.py` — корутины (`asyncio.gather`);
- `run_all.py` — запуск всех + последовательный baseline + таблица.

### Результаты (N = 10⁸, 8 воркеров; CPU 10 ядер, macOS)

| Подход | Время, c | Ускорение |
|---|---:|---:|
| sequential | 1.999 | 1.00× |
| **multiprocessing** | **0.501** | **3.99×** |
| threading | 2.019 | 0.99× |
| async | 1.990 | 1.00× |

### Анализ
- **multiprocessing** дал ускорение ~4× — процессы реально работают параллельно
  на разных ядрах. Ускорение меньше числа воркеров (8) из-за накладных расходов
  на создание процессов и сериализацию результатов.
- **threading** ускорения **не дал** (даже чуть медленнее): из-за GIL потоки с
  CPU-нагрузкой выполняются по очереди, добавляя накладные расходы на
  переключение.
- **async** ускорения **не дал**: корутины выполняются в одном потоке
  кооперативно, а внутри вычислений нет точек `await`, в которых можно было бы
  переключиться. Фактически — последовательное выполнение.

**Вывод:** для CPU-bound задач подходит только `multiprocessing`.

## Задача 2. Параллельный парсинг веб-страниц с сохранением в БД

Каждая программа функцией `parse_and_save(url)` загружает HTML, извлекает
заголовок `<title>` (стандартным `html.parser`, без BeautifulSoup) и сохраняет
его в **базу данных из ЛР1** (`LR1/hackathon.db`) — в добавленную туда таблицу
`parsed_pages` (поля: `url`, `title`, `approach`, `parsed_at`). Доменные таблицы
ЛР1 не затрагиваются.

Парсится список из 12 страниц (`urls.py`); список целиком обрабатывается
параллельно — по одной подзадаче на URL.

### Файлы
- `db.py` — подключение к БД из ЛР1, модель `ParsedPage`, синхронная `save_page()`
  (для threading/multiprocessing) и **асинхронная `async_save_page()`** (для async,
  SQLAlchemy async + драйвер `aiosqlite`);
- `urls.py` — список URL, извлечение `<title>`, синхронная загрузка с поддержкой gzip;
- `threading_parse.py` — потоки + `urllib`;
- `multiprocessing_parse.py` — процессы (`Pool`) + `urllib`;
- `async_parse.py` — `asyncio` + `aiohttp`; **запись в БД тоже асинхронная** (`await async_save_page`);
- `run_all.py` — запуск всех + последовательный baseline + таблица.

> Асинхронная запись: в async-подходе и сетевой запрос, и сохранение в БД
> выполняются неблокирующе (через `await`), не занимая событийный цикл — вместо
> прежней обёртки `asyncio.to_thread` вокруг синхронного `save_page`.

### Результаты (12 страниц; macOS)

| Подход | Время, c | Ускорение |
|---|---:|---:|
| sequential | 15.193 | 1.00× |
| **async** | **1.285** | **11.83×** |
| threading | 2.162 | 7.03× |
| multiprocessing | 2.299 | 6.61× |

### Анализ
- Все три параллельных подхода дали значительное ускорение — задача **I/O-bound**
  (большую часть времени программа ждёт ответ сервера).
- **async** — самый быстрый и самый лёгкий: один поток обслуживает все
  соединения, пока корутины ждут сеть (`await`), управление передаётся другим.
  Нет накладных расходов на потоки/процессы.
- **threading** немного медленнее async, но тоже эффективен: во время сетевого
  ожидания GIL освобождается, и потоки фактически работают одновременно.
- **multiprocessing** — самый медленный из параллельных: создание процессов,
  сериализация и отдельное подключение к БД в каждом процессе создают накладные
  расходы, которые для сетевого ввода-вывода не окупаются.

**Вывод:** для I/O-bound задач лучше всего подходят `asyncio` и `threading`;
`multiprocessing` для них избыточен.

## Итоговое сравнение

| | CPU-bound (задача 1) | I/O-bound (задача 2) |
|---|---|---|
| **threading** | ✗ нет ускорения (GIL) | ✓ хорошо |
| **multiprocessing** | ✓ лучший (≈ число ядер) | ~ работает, но дорого |
| **asyncio** | ✗ нет ускорения | ✓ лучший |

> Цифры времени зависят от железа и состояния сети, поэтому при перезапуске будут
> немного отличаться, но **соотношение** между подходами сохраняется.
